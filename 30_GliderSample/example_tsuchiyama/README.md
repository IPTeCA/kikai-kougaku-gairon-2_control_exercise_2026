# example_tsuchiyama（元: 90_tsuchiyama）

元ファイル：`SampleProgram/90_tsuchiyama/glider_ta.ino`（+ `hz_sleep.h`）

## これは何？

投げ上げ/滑空のような **実機条件**を強く意識した姿勢制御の例です。

主な特徴は次の通りです。

- 角度（Madgwickの roll/pitch/yaw）と角速度（ジャイロ）を使って制御量を作る
- サーボの更新周期を落として（マルチレート）安定させる
- 飛行状態の推定（加速度しきい値）で制御の振る舞いを切り替える

## 挙動（ざっくり）

1. 起動後、`calc_offset()` で姿勢角のオフセット（ゼロ点）を平均して求める
2. ループで IMU → Madgwick 更新 → `ang_`（オフセット差し引き後の角度）を計算
3. roll/pitch の制御量（`pid_roll`, `pid_pitch`）を計算してサーボへ反映

## なぜ「工夫」が必要？

### 1) D項は「角度差分」より「ジャイロ」を使う

角度差分で微分を作るとノイズで暴れやすいので、角速度（ジャイロ）をそのままD成分として使っています。

### 2) I項は常時ONにしない（段階的に解放）

強い加速度が乗る区間では角度推定が乱れやすく、I項を回すと溜まりやすいので、

- ある条件（`LIMIT_ACCX` を超えたら `gone=true`）までは I成分を実質OFF
- 条件成立後に I成分を上限付きで解放（`limit_i` と `constrain()`）

という扱いになっています。

### 3) サーボは毎ループ更新しない（マルチレート）

IMU更新は高頻度（`MEASURING_FREQ`）でも、サーボ書き込みは `SERVO_SKIP` で間引いています。
サーボの応答や機構の制約を考えると、更新を落とした方が安定する場合があります。

## 読み方（おすすめ）

1. `calc_offset()`：なぜオフセットが必要か（ゼロ点、取り付け誤差）
2. `gone` / `LIMIT_ACCX`：なぜ飛行状態で挙動を切り替えるか
3. `pid_roll` / `pid_pitch`：角度（P）と角速度（D）、I成分の扱い
4. `sv*.write(constrain(...))`：サーボの範囲制約とオフセット
